今日のデイリーリサーチのテーマを2つ選定してください。

## 手順

1. `config.toml` を読み、各トラック（tech / personal）の対象ドメインとスコアリング基準を把握する
2. `past_topics.json` を読み、過去30日以内のテーマと重複しないよう確認する
3. WebSearch で最新トレンドを調査する
4. config.toml のスコアリング基準で候補を評価し、各トラック最高スコアのテーマを1つずつ選定する

## 出力形式

JSON のみを出力すること。説明文やマークダウンは一切含めない。

## 設計メモ: モデル配置の根拠 (2026-02-20)

### 現行: Opus がテーマ選定を一括担当

検索クエリ設計・WebSearch 実行・スコアリング・選定を Opus が一括で行う。

### 検討・棄却した代替案

#### 案A: Opus クエリ設計 → Haiku 検索 → Opus スコアリング

Opus に検索クエリ設計だけさせ、安価な Haiku で WebSearch を実行する案。

**棄却理由:**

1. **Opus はクエリ設計にオーバースペック** — 検索クエリ設計は「広さ」の仕事であり「深さ」の仕事ではない。config.toml のドメインリストから検索クエリを作るのに Opus の推論力は不要
2. **Haiku の要約品質が Pass 1 のボトルネックになる** — Pass 0 の本質は検索結果の要約・構造化にある。Haiku ではトレンドのニュアンスや「なぜ今話題か」の文脈が薄くなり、Opus のスコアリング素材の質が下がる
3. **`claude -p` 1回追加のオーバーヘッド** — 起動 ~5-10秒、フォールバック分岐の複雑化、追加のプロンプト・バリデーションが必要
4. **コスト削減効果が微小** — Pass 0+1 の合計はパイプライン全体の ~17% ($0.30/$1.80)。内部構成を最適化しても全体コストへのインパクトは小さい

#### 案B: 3パス化 (Sonnet 検索 → Opus 判断 → Sonnet 執筆)

`docs/plans/THREE-PASS-PLAN.md` 参照。検索を Sonnet に分離し、Opus は純粋な判断のみ行う案。Opus のタイムアウト問題への対策として有効。
